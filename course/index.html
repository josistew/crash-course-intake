Chapter 2

Hooks Deep Dive

Hooks are commands that run automatically at specific points in Claude's workflow. Think of them as "if Claude does X, also do Y."

You configure them in .claude/settings.json.

Four Hook Types

  • Command hooks — Run a shell command (formatting, linting, logging)
  • HTTP hooks — POST JSON to a URL (send notifications, trigger webhooks)
  • Prompt hooks — Ask an LLM a yes/no question (smart gatekeeping)
  • Agent hooks — Spawn a subagent to verify complex conditions

Five Hook Events

EventWhen It Fires
PreToolUseBefore Claude runs any tool (block dangerous commands)
PostToolUseAfter a tool completes (auto-format files, log actions)
PostToolUseFailureWhen a tool fails (custom error handling)
PermissionRequestWhen Claude asks for permission (auto-approve safe patterns)
NotificationWhen Claude sends a notification (forward to Slack, Telegram)

Auto-format on every file write

settings.json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "type": "command",
        "command": "prettier --write $CLAUDE_FILE_PATH"
      }
    ]
  }
}

Send Telegram notification when Claude finishes

settings.json
{
  "hooks": {
    "Notification": [
      {
        "type": "http",
        "url": "https://api.telegram.org/bot<TOKEN>/sendMessage",
        "method": "POST",
        "body": {
          "chat_id": "<YOUR_CHAT_ID>",
          "text": "Claude finished: $CLAUDE_NOTIFICATION"
        }
      }
    ]
  }
}

Block destructive git commands

settings.json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "type": "prompt",
        "prompt": "Is this command destructive (force push, reset --hard, rm -rf)? Answer YES to block, NO to allow."
      }
    ]
  }
}

Chapter 3

MCP Servers Deep Dive

MCP (Model Context Protocol) lets Claude connect to external tools and data sources. Each MCP server exposes "tools" that Claude can call like native capabilities.

Where MCP Configs Live

  • Global: ~/.claude/settings.json — available in all projects
  • Project: .claude/settings.json — project-specific, committed to git
  • Local: .claude/settings.local.json — project-specific, NOT committed (good for secrets)

Config structure

settings.json
{
  "mcpServers": {
    "server-name": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/path/to/folder"],
      "env": {}
    }
  }
}

MCP Servers You Should Know

ServerWhat It DoesWhy You'd Use It
@anthropic/claude-code-mcpExposes Claude Code itself as an MCP toolLet Cowork or other Claude instances call Claude Code
server-filesystemRead/write files in specified directoriesGive Claude access to specific folders
server-githubGitHub API (issues, PRs, repos)Manage repos without leaving Claude
mcp-server-fetchFetch web pagesLet Claude read URLs
Google Sheets MCPRead/write Google SheetsYour LeaseJenny dashboard, lead tracking
Slack MCPSend/read Slack messagesNotifications, team updates

MCP tools in hooks

MCP tools follow the pattern mcp__<server>__<tool>. You can write hooks that trigger on specific MCP tools:

settings.json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "mcp__github__.*",
        "type": "command",
        "command": "echo 'GitHub action taken' >> ~/claude-audit.log"
      }
    ]
  }
}

Skills & Slash Commands

You use some skills already but want more. Here's the full picture.

Built-in Skills You Might Not Know

  • /simplify — Reviews changed code for reuse, quality, efficiency, then fixes issues
  • /batch — Run the same operation across multiple files
  • /copy — Interactive picker to copy specific code blocks (NEW)
  • /memory — View and manage auto-memory files
  • /model — Switch models mid-conversation (now shows current model)
  • /voice — Voice input mode (NEW, rolling out gradually)

Custom Skills

Custom skills live in .claude/skills/ as markdown files. Each file becomes a slash command.

Skill file structure

.claude/skills/my-skill.md
# /my-skill-name

Description of what this skill does.

## Instructions

Step-by-step instructions Claude follows
when this skill is invoked.

## Context

Any context Claude needs (file paths,
conventions, etc.)

Example: /deploy skill

.claude/skills/deploy.md
# /deploy

Deploy the current project to Vercel.

## Instructions

1. Run the build command to verify no errors
2. Check git status — commit any uncommitted
   changes first
3. Run `vercel --prod` to deploy
4. Report the deployment URL when done

## Context

- Always use `--prod` flag for production deploys
- If build fails, fix the issue before deploying

Example: /morning-report for LeaseJenny

.claude/skills/morning-report.md
# /morning-report

Generate a morning leads report.

## Instructions

1. Read the Google Sheet for LeaseJenny leads
2. Count new leads from the last 24 hours
3. Summarize qualification status
4. List any leads that need follow-up
5. Format as a clean summary I can skim
   in 30 seconds

Chapter 5

Subagents

Subagents let Claude spawn independent workers that run in parallel. This is one of the most powerful features you're not touching.

When to Use Subagents

  • Researching multiple things at once (search 3 codebases simultaneously)
  • Running independent tasks that don't depend on each other
  • Protecting your main context window from getting bloated with search results
  • Exploring a codebase deeply without losing your place

How They Work

Claude automatically uses the Agent tool to spawn subagents. You can also configure agent types in your CLAUDE.md or skills.

Built-in Agent Types

TypePurpose
ExploreFast codebase exploration — find files, search code, answer questions
PlanArchitecture planning — design implementation strategies
general-purposeResearch, search, multi-step tasks

Example prompt that triggers subagents

Try This

"Research how Square's API handles webhook notifications, AND check my LeaseJenny codebase for any existing webhook handlers, AND find examples of Square webhook implementations on GitHub"

Claude will spawn 3 parallel agents instead of doing these sequentially.

Your Opportunity

When working on multi-project tasks (LeaseJenny + stock screener + Humanity), subagents let you research across all of them simultaneously.

Hands-On Exercises
Ex 5.1 Multi-Part Research Prompt

   skills on GitHub

Chapter 7

The Creator’s Playbook

Boris Cherny (creator of Claude Code) shared his 12-point playbook for getting maximum output from the tool. These aren’t theoretical — they’re the patterns he uses daily building Claude Code itself.

Philosophy

“Claude Code is not an autocomplete tool. It’s a junior developer with perfect memory and infinite patience. Treat it like one: give it clear briefs, review its work, and build systems that compound.”

The 12 Practices

1. Run Parallel Sessions

Open multiple terminal tabs, each running a separate Claude Code instance on a different task. While one refactors a module, another writes tests, and a third researches an API. They share the same CLAUDE.md but don’t block each other.

3 terminals, 3 tasks
# Tab 1
claude "Refactor the auth middleware to use JWT"

# Tab 2
claude "Write integration tests for the /users endpoint"

# Tab 3
claude "Research the Stripe API for subscription billing"

2. Use Opus with Extended Thinking

For architecture decisions and complex debugging, switch to Opus with thinking enabled. The thinking budget gives Claude space to reason through trade-offs before committing to an approach.

Switch mid-session
/model opus
# Now ask your hard question
"Design the data model for multi-tenant billing
with usage-based pricing. Think through the trade-offs."
Tip

Use Sonnet for routine tasks (edits, tests, boilerplate) and Opus for decisions that shape the architecture. Cost-effective and higher quality where it matters.

3. Shared CLAUDE.md as Single Source of Truth

Your CLAUDE.md file is the project’s constitution. Every session reads it. Put coding conventions, architecture decisions, file structure, and “never do this” rules there. When Claude does something wrong, don’t just fix it — update CLAUDE.md so it never happens again.

CLAUDE.md (project root)
# Architecture
- All API routes go in src/routes/
- Use Zod for all request validation
- Never use any as a TypeScript type

# Conventions
- Tests go next to source files (foo.test.ts)
- Use named exports, not default exports
- Error handling: throw AppError, never raw strings

# DO NOT
- Do not modify the database schema without asking
- Do not install new dependencies without approval

4. @claude on Pull Requests

Tag @claude in GitHub PR comments to get automated code review. Claude reads the diff, checks against your CLAUDE.md conventions, and leaves line-by-line feedback. It catches what linters miss — logic bugs, missing edge cases, convention violations.

Tip

Set up the GitHub integration so every PR automatically gets a Claude review pass. It’s like having a senior engineer who never gets tired of reviewing.

5. Plan Mode Before Complex Work

For any task touching 3+ files, use plan mode first. Claude explores the codebase, designs an approach, and presents it for your approval before writing a single line of code. Catches wrong assumptions early.

Plan, then execute
/plan
"Add rate limiting to all API endpoints.
Consider Redis vs in-memory, per-user vs per-IP,
and how it interacts with our existing auth middleware."

# Review the plan, adjust, then approve

6. Configure /permissions for Flow State

Use /permissions to auto-allow safe operations so Claude doesn’t interrupt you for every file read or test run. Keep approval required for destructive operations (git push, file deletion, package installs).

Example permission setup
# Auto-allow:
Read, Glob, Grep, WebSearch        # research
npm test, npm run lint              # validation
git status, git diff, git log       # read-only git

# Require approval:
git push, git reset                 # destructive git
rm, unlink                          # deletions
npm install                         # dependency changes

7. PostToolUse Hooks for Auto-Validation

Run validation automatically after Claude edits code. A PostToolUse hook on Write/Edit can run your linter, type checker, or tests immediately — catching issues before they compound.

.claude/settings.json
{
  "hooks": {
    "PostToolUse": [
      {
        "type": "command",
        "matcher": "Write|Edit",
        "command": "npx tsc --noEmit 2>&1 | head -20"
      }
    ]
  }
}

If the type checker fails, Claude sees the errors immediately and self-corrects. No manual feedback loop needed.

8. Modular Subagents for Parallel Work

Break complex tasks into subagent calls. Each subagent gets a focused scope, runs in parallel, and returns results. Think of it as delegating to a small team instead of one overwhelmed developer.

Prompt pattern
"For this migration:
1. Spin up an Explore agent to map all usages of the old API
2. Spin up a second agent to research the new API's interface
3. Then use the results to build the migration plan"

9. Build Custom Slash Commands

Create /commands for workflows you repeat. A /deploy that runs tests, builds, and pushes. A /morning that checks all your projects. A /review that audits the current diff against your standards.

.claude/commands/deploy.md
Run the full deploy pipeline:
1. Run all tests (fail fast)
2. Build for production
3. If everything passes, deploy to Vercel
4. Report the deploy URL when done

10. Git Worktrees for Isolation

Use git worktrees to run parallel Claude sessions on the same repo without conflicts. Each worktree is an isolated copy — one session can refactor main while another builds a feature branch, without file lock collisions.

Set up a worktree
# Create a worktree for a feature branch
git worktree add ../my-repo-feature feature-branch

# Run Claude in the worktree
cd ../my-repo-feature && claude

# Or use Claude's built-in worktree support
/worktree

11. Verification Loops

Always have Claude verify its own work. After any significant change, ask it to review what it did, run tests, and confirm the change matches your intent. Build this into your slash commands and prompts.

End every task with
"Before marking this done:
1. Run the test suite
2. Check for TypeScript errors
3. Review the diff and confirm it matches the original request
4. List any edge cases you're unsure about"

12. Stop Hooks as Guardrails

Use Stop hooks to prevent Claude from making unwanted changes. A hook can block commits to protected branches, prevent deletion of critical files, or enforce naming conventions before changes are saved.

Prevent accidental main pushes
{
  "hooks": {
    "PreToolUse": [
      {
        "type": "command",
        "matcher": "Bash",
        "command": "if echo \"$CLAUDE_TOOL_INPUT\" | grep -q 'push.*main'; then echo 'BLOCKED: No pushing to main'; exit 2; fi"
      }
    ]
  }
}

Exit code 2 blocks the action entirely. Claude sees the message and adjusts.

Chapter 9

Cowork — What It Is & Setup

Claude Code's power applied to knowledge work — not just coding. It runs in the Claude Desktop app.

What Is Cowork?

The shift: Instead of chatting back and forth, you describe an outcome, Claude plans it, executes it, and you come back to finished work. It's like leaving a task for a coworker.

What it can do

  • Read, edit, and create files on your computer
  • Generate polished deliverables (spreadsheets with formulas, presentations, reports)
  • Browse the web (with Claude in Chrome extension)
  • Connect to external tools (Google Drive, Gmail, Slack, etc.)
  • Run long tasks while you step away
  • Coordinate multiple sub-agents in parallel
  • Schedule recurring tasks with /schedule

Setup (5 Minutes)

  1. Open Claude Desktop (make sure it's updated)
  2. Click the "Cowork" tab in the mode selector (next to "Chat")
  3. You're in. That's it.

Give it folder access

  • When you start a task involving files, Cowork will ask which folder to access
  • Grant access to specific folders (e.g., ~/LeaseJenny, ~/humanity-brand)
  • Claude can only see what you explicitly allow
